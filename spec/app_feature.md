## 0. 文档职责

本文是 STRIDE 手机端 v1 的功能范围、信息架构、页面职责和状态覆盖基线。它用于约束：

- Stitch 手机端页面设计与 screen inventory；
- Flutter 路由、页面和用户工作流；
- 手机端 API 契约与 Deep Link；
- 产品验收、视觉验收和发布完整性检查。

发生冲突时，优先级为：

1. 本文中的 v1 `HARD` 决策；
2. `mobile/DESIGN.md` 与 `mobile/STRIDE_OVERRIDES.md` 中不冲突的视觉规则；
3. 当前 Stitch 正式 screen；
4. Flutter 当前实现。

Stitch 是正式 UI 设计源。Flutter 当前已有页面不自动代表产品决策正确；当实现、Stitch 与本文不一致时，应先按本文更新 Stitch，再落地 Flutter。

### 0.1 用户可见术语

| 概念 | 正式用词 | 禁止作为用户主文案 |
|---|---|---|
| 长周期训练地图 | **赛季训练计划** | 训练总纲、Master Plan、Draft |
| 某一自然周的课表 | **本周课表** | 单周计划、Weekly Plan、周计划 Draft |
| 一次具体训练安排 | **训练课** | session、workout item |
| AI 对话入口 | **教练** | QA、Agent、Chatbot |
| 训练负荷差值 | **状态储备（Form）**，详情中可解释为 chronic - acute | 直接裸露 TSB 而无解释 |

内部 API、代码与数据库可以保留既有英文命名，但不得直接泄漏到用户界面。

## 1. 产品主线

### 1.1 一句话故事

> 注册 → 绑定手表 → 同步历史 → 立即看到活动与状态 → 主动创建赛季训练计划 → 审阅并启用 → 生成并审阅本周课表 → 推送到手表 → 查看今日训练 → 完成后看 AI 点评并填 RPE → 周复盘 → 生成下周课表 → 持续循环

### 1.2 核心价值原则

1. **Onboarding 后立即有价值**：即使用户不创建计划，也能看到最近 90 天活动、基础状态和 AI 点评。
2. **训练计划由用户主动触发**：绑定手表不等于必须立刻创建赛季训练计划。
3. **手表是 v1 核心前提**：没有有效手表绑定时，训练、数据和教练的个体化能力不可用。
4. **对话即统一入口**：用户可在教练中查询状态、复盘训练、创建或调整计划。
5. **写操作先提案、后确认**：计划、目标、课表等任何修改都必须由用户确认后生效。
6. **建议必须可解释**：训练建议需展示关键依据，不能只给结论。
7. **不伪造数据**：缺少手表数据、校准指标或外部数据时，明确显示缺失和补齐方式。

### 1.3 v1 核心闭环

v1 发布必须打通以下闭环，不能只存在孤立页面：

1. 手表同步后的新活动可打开详情；
2. 活动详情可填写或修改训练反馈；
3. 反馈进入周复盘和下一周生成上下文；
4. 本周课表可查看训练课详情并推送手表；
5. 教练提出的计划变更可审阅、确认并应用；
6. 赛季训练计划、本周课表和今日训练之间可双向到达；
7. 推送通知能 Deep Link 到对应活动、反馈、课表、复盘或生成结果。

## 2. v1 范围决策（HARD）

### 2.1 视觉与平台

1. **主色**：STRIDE Green 为 `#1FAD5B`。旧色 `#00E676` 不再用于正式设计 token。
2. **平台**：Android 优先；布局同时保证未来 iOS 可复用。
3. **目标画布**：手机单列布局，标准设计宽度 390 logical px，并适配更小屏幕。
4. **数字字体**：配速、心率、距离、时长、日期、负荷、百分比和组次统一使用等宽数字。
5. **设计约束**：主色只用于主操作、选中态、焦点和有意义的状态，不作为大面积背景或装饰渐变。

### 2.2 底部导航

v1 固定为四个一级 Tab：

| 顺序 | Tab | 根页面 | 用户任务 |
|---|---|---|---|
| 1 | **跑者** | 跑者主页 | 今天状态如何、今天练什么、最近练得怎样 |
| 2 | **训练** | 训练中心 | 管理赛季训练计划、本周课表、历史与复盘 |
| 3 | **数据** | 数据概览 | 查看健康、负荷、能力、预测和 PB |
| 4 | **教练** | 教练会话 | 提问、诊断、复盘、创建或调整计划 |

约束：

- 四个 Tab 等宽，不设置中间凸起按钮；
- 一级 Tab 切换后保留各自导航栈和滚动位置；
- 全屏详情页不重复显示底部导航；
- **发现不占底部 Tab**；
- **个人中心不占底部 Tab**。

### 2.3 侧边菜单

四个一级页面左上角统一提供侧边菜单入口。菜单顺序为：

1. 用户身份与手表状态；
2. 个人中心；
3. 发现；
4. 手表管理；
5. 同步手表数据；
6. 通知设置；
7. 意见反馈；
8. 常见问题；
9. 检查更新；
10. 关于 STRIDE；
11. 退出登录。

个人中心、发现和设置类页面通过侧边菜单进入，不得挤占训练主线的一级导航。

### 2.4 设备与记录

- COROS 是 v1 正式支持设备；Garmin 显示为 v1.1 能力，不能伪装为可用。
- v1 不提供手机 GPS 实时跑步记录，因此不设置“记录”Tab。
- 活动记录来自手表同步；用户可在跑者主页进入完整活动列表。
- 解绑最后一块手表后，立即进入未绑表拦截状态。

### 2.5 发现与社交

- v1 保留当前“发现”页面，作为侧边菜单中的二级入口。
- v1 只承诺展示用户已加入的跑团列表及相应空状态。
- 搜索、创建、加入、跑团详情、动态流、评论、关注和公开主页不属于 v1 核心范围。
- 未实现的社交动作不得伪装成可完成的主 CTA。

### 2.6 营养

营养属于完整训练服务的一部分，在 v1 实现，但不占一级 Tab。入口来自训练中心、训练课详情和个人中心。

### 2.7 v1 Beta 权益

- v1 采用邀请码 Beta；已获邀请且完成注册的用户可使用本文定义的核心能力；
- v1 不展示 Free、Pro、Elite 权益差异，不设置订阅墙或支付入口；
- 本条是产品愿景中商业化路线的阶段性覆盖，不改变后续订阅方向；
- 未实现的付费入口不得以禁用 CTA 或虚假价格干扰 Beta 用户。

## 3. 全局状态模型与不变量

### 3.1 用户门控状态

| 状态 | 应用行为 |
|---|---|
| Auth 正在恢复 | 显示启动加载，不短暂暴露受保护页面 |
| 未登录 | 进入身份认证流程 |
| 已登录但 Onboarding 未完成 | 返回对应 Onboarding 步骤 |
| 已完成但无手表 | 全屏未绑表拦截，只允许绑定或退出 |
| 已登录且手表有效 | 进入请求页面，默认进入跑者 Tab |

### 3.2 训练状态

赛季训练计划和本周课表是两个正交生命周期，不能压缩成一个三值字段后丢失生成、审阅和失败状态。

| 生命周期 | 状态 | 含义 |
|---|---|---|
| `season_state` | `none` | 尚无赛季训练计划 |
| `season_state` | `generating` | 后台生成中 |
| `season_state` | `review_ready` | 草稿已完成，等待用户审阅和启用 |
| `season_state` | `active` | 已启用，约束后续本周课表 |
| `season_state` | `failed` | 生成或恢复失败，可重试 |
| `season_state` | `completed` | 目标周期结束，只读归档，可创建新目标 |
| `week_state` | `not_applicable` | 尚无已启用赛季训练计划 |
| `week_state` | `none` | 当前目标周尚无课表 |
| `week_state` | `generating` | 本周课表生成中 |
| `week_state` | `review_ready` | 草稿已完成，等待审阅和启用 |
| `week_state` | `active` | 本周课表已启用，可推送和执行 |
| `week_state` | `failed` | 生成失败，可重试 |
| `week_state` | `completed` | 该周结束，进入复盘和历史 |

跑者主页和训练中心至少覆盖以下组合：

| 状态组合 | 主任务 |
|---|---|
| `none / not_applicable` | 创建赛季训练计划 |
| `generating / not_applicable` | 查看赛季生成进度 |
| `review_ready / not_applicable` | 审阅赛季训练计划 |
| `active / none` | 生成本周课表 |
| `active / generating` | 查看本周课表生成进度 |
| `active / review_ready` | 审阅并启用本周课表 |
| `active / active` | 查看今日训练和本周进度 |
| 任一 `failed` | 解释失败、保留原有效计划并重试 |
| `completed / not_applicable` | 查看归档并创建新目标 |

API 可保留兼容性的派生 `plan_state`，但 UI 不得只依赖它隐藏上述生命周期。

### 3.3 写操作契约

任何计划或目标写操作遵循：

`用户意图 → 必要追问 → 生成提案 → 展示差异与影响 → 用户确认 → 应用 → 成功或可恢复失败`

HARD 约束：

- AI 回复不等于已修改；
- 伤病、疼痛、明显身体不适不得自动修改计划；
- 用户拒绝、离开或网络失败时，原计划保持不变；
- 提案必须绑定明确对象：赛季训练计划、本周课表或某训练课；
- 应用前发现对象版本已变化时，必须提示重新生成，不能覆盖新版本。

### 3.4 数据与时间

- 所有面向用户的日、周和“今天”按 `Asia/Shanghai` 分类；
- 用户可见数据标注来源：手表原始、STRIDE 计算、AI 建议；
- 过期缓存显示最后更新时间和“数据可能已过期”；
- 下拉刷新只重新请求当前页面数据；显式“同步手表数据”才触发完整同步；
- 同步过程全局单实例，重复点击不能创建并发任务。

## 4. 模块 A：身份认证

### A0. 冷启动与 Auth 恢复

- 有有效 access token：进入目标 Deep Link 或跑者主页；
- access token 过期但 refresh token 有效：静默刷新一次；
- refresh 失败：清理会话并进入登录，显示“登录已过期”；
- Auth 恢复期间显示稳定启动页，不闪现主页或登录页；
- 网络异常与凭据错误必须使用不同文案。

### A1. 身份入口

- 展示 STRIDE 品牌、价值说明、登录与注册两个入口；
- 已登录用户不得停留在此页；
- 不在此页要求创建训练计划。

### A2. 登录

- 邮箱 + 密码；
- 提交中禁用重复操作；
- 区分账号密码错误、网络异常、服务异常；
- 提供“忘记密码”入口；
- 登录成功后恢复原 Deep Link，否则进入跑者主页。

### A3. 注册

- 邮箱、密码、确认密码、邀请码、协议勾选；
- 实时校验邮箱格式、密码至少 8 位、两次密码一致、邀请码非空；
- 用户协议和隐私政策必须可打开，不使用不可点击的装饰文字；
- 注册成功进入 B1，不直接进入主应用。

### A4. 找回密码

- 输入注册邮箱并请求重置邮件；
- 无论账号是否存在都使用相同成功文案，避免泄露注册状态；
- 邮件 Deep Link 打开“设置新密码”，支持冷启动和 Auth 路由恢复；
- 新密码至少 8 位并二次确认；
- token 过期、已使用、无效和网络失败均提供明确恢复方式；
- 重置成功后返回登录，不自动复用旧会话。

## 5. 模块 B：新手引导

新手引导共四个正式步骤。手表绑定不可跳过，引导结束后进入跑者主页。

### B1. 选择手表品牌

- COROS 可选择；
- Garmin 显示“v1.1 即将支持”并禁用；
- 不提供跳过；
- 说明绑定手表是个体化建议的前提。

### B2. 绑定 COROS

- 输入 COROS 邮箱和密码；
- 自动检测全球、中国、欧洲区域，并允许手动切换；
- 明确说明绑定阶段只读取训练数据；后续仅在用户主动确认“推送到手表”时写入训练课，不修改其他手表设置；
- 区分区域不匹配、账号密码错误、网络错误和服务错误；
- 输入内容在失败后保留，密码按安全策略处理。

### B3. 首次同步

- 拉取最近 90 天活动和健康数据；
- 展示当前阶段、已同步活动数、健康数据天数和总体进度；
- 同步任务由服务端持有，App 重启后可恢复进度；
- 同步失败提供重试，不丢失已完成部分；
- 同步未完成前不进入 B4。

### B4. 基础信息

- 性别、出生年、身高、体重；
- 静息心率优先使用手表建议值，允许修改；
- 最大心率优先使用个体校准值，其次才使用年龄公式建议值；
- 明确区分“建议值”和“用户填写值”；
- 完成后进入跑者主页，不强制创建赛季训练计划。

### B5. 未绑表拦截

适用于旧用户、解绑用户和绑定状态失效用户：

- 训练、数据和教练的个体化能力不可访问；
- 全屏说明原因和已有数据是否保留；
- 主 CTA 为“立即绑定手表”；
- 可进入受限个人中心处理协议、隐私、支持、注销账号和退出登录；
- 不允许通过返回键绕过。

## 6. 模块 C：赛季训练计划

赛季训练计划是长期训练地图，定义目标周期、阶段、周量方向、关键训练和里程碑。它是本周课表的上位约束，但不会在启用时自动创建本周课表。

### C1. 训练目标

- 目标类型：备赛、提升 PB、减脂、健康跑、维持；
- 备赛进一步采集比赛日期、距离和可选目标成绩；
- 提升 PB 必须选择目标距离和可选目标成绩；
- 减脂、健康跑和维持必须选择目标周期；默认建议 12 周，允许 8-24 周；
- 每周可训练天数 3-6 天；
- 按星期采集可训练与不可训练日期；每个可训练日可选择早、午、晚时段；
- 力量训练意愿：愿意、不愿意、看情况；
- 编辑已有目标时，进入变更提案流程，不直接覆盖现有赛季计划。

### C2. 跑步档案（可选）

- 跑龄、当前周跑量；
- 5K、10K、半马、全马 PB，可全部留空；
- 伤病史与“暂无”；
- 可跳过并明确说明将使用哪些默认值；
- 已有档案时预填，并允许在个人中心再次编辑。

### C3. 三年历史数据准备

- 按需同步最近三年跑步活动；
- 展示活动数、年份区间、百分比和当前阶段；
- 支持服务端断点续传和 App 重启恢复；
- 失败可重试，不重复下载已完成部分；
- 数据不足时说明计划置信度影响，允许用户继续。

### C4. 生成赛季训练计划

- 生成是分钟级后台任务；
- 阶段文案：读取历史 → 评估体能 → 结合目标 → 规划周期 → 生成计划；
- 支持 queued、running、ready、failed、interrupted 状态；
- 用户可以离开，完成后通过通知召回；
- 重复点击不会创建多个相同任务；
- 返回应用后显示“计划已准备好”，由用户主动进入审阅。

### C5. 审阅与启用

- 展示周期、阶段、周量范围、关键训练、里程碑、主要风险和安排理由；
- 教练可追问缺失信息；
- 用户可用自然语言要求修改；
- 修改结果以旧 → 新差异展示，明确阶段、周量和里程碑影响；
- 最终唯一主 CTA 为“启用计划”；
- 启用成功后进入 C6；
- 启用不自动生成本周课表。

### C6. 当前赛季训练计划

常驻展示：

- 起止日期、总周数、当前阶段和当前周；
- 比赛倒计时或目标周期进度；
- 阶段时间轴，区分已完成、当前、未来；
- 每阶段的日期、重点、周量区间、关键训练和里程碑；
- 已完成里程碑的实际结果；
- “与教练调整计划”和“调整历史”入口；
- 本周尚无课表时，提供“生成本周课表”入口。

### C7. 调整赛季训练计划

- 顶部固定展示当前目标和阶段摘要；
- 支持目标、比赛日期、训练时间、伤病和生活安排变化；
- 教练先澄清，再返回差异提案；
- 用户可继续追问、拒绝或确认整个提案；
- 应用前说明对已生成本周课表和已推送手表训练的影响；
- 已推送训练若会失效，明确提示需要用户在手表端处理的内容；
- 无有效变化和教练拒绝不合理请求是正式状态，不伪装为成功。

### C8. 调整历史与版本

- 时间线展示修改时间、原因、发起方式和摘要；
- 可查看任一版本快照；
- 历史版本只读；恢复旧版本也必须走提案和确认流程。

### C9. 版本比较

- 从 C8 选择两个版本后进入；
- 对比目标、日期、阶段、周量、关键训练和里程碑；
- 只展示实际变化，未变化区块默认折叠；
- 比较结果只读；“以此版本为基础调整”进入 C7，而不是直接恢复。

## 7. 模块 D：训练与日常主循环

### D0. 训练中心（训练 Tab 根页面）

训练中心不是单纯的周列表，必须同时承载长期计划和当前执行：

- 赛季训练计划摘要与入口；
- 当前阶段和目标；
- 全赛季周训练量柱状图：按周展示预计跑量，阶段分组着色，当前周高亮；
- 当前周、本周完成率和下一个训练课；
- 下一周草稿或生成条件；
- 每日营养建议入口；
- 历史周列表与周复盘入口；
- 无赛季计划、已有赛季但无本周课表、已有本周课表三种正式状态；
- 赛季计划与本周课表均可从此稳定进入。

周训练量图的移动端约束：

- 位于赛季训练计划摘要和本周课表之间；
- 展示整个赛季的预计周跑量，不只展示当前阶段；
- 每周一根细柱，按基础、进展、赛前、减量、比赛等阶段使用低饱和色分组；
- 当前周使用 STRIDE Green 描边，并显示 `Wxx · 当前`；
- 横轴只标阶段起点或关键周，避免 20 个周标签挤压；
- 纵轴不显示密集刻度，只保留 `预计周跑量 · km/周` 与当前周数值；
- 图表用于快速理解训练量节奏，点击或“查看计划”进入 C6 查看完整赛季详情；
- 不与单独阶段圆点轨道同时展示，避免重复表达。

### D1. 生成本周课表

- 基于已启用赛季训练计划、当前状态、上周完成情况和训练反馈；
- 手动触发：用户可从跑者主页、训练中心和周复盘生成当前周或下一周草稿；
- 自动触发：每周日 23:59 在至少完成 1 节训练且提交至少 1 条反馈时，为下一自然周生成 `review_ready` 草稿；
- 自动生成不自动启用、不自动推送手表；完成后通知用户审阅；
- 不满足自动生成条件时不静默降级，而是提醒用户补充反馈或手动确认降级生成；
- 首次生成没有上周反馈时，明确显示“使用基础档案和当前状态生成”；
- 后续缺少反馈时，提示影响；允许继续时明确标识为降级生成；
- 秒级任务展示生成进度并防止重复提交；
- 已存在该周的 `active` 课表时，不允许直接覆盖；应进入 D4 生成调整提案；
- 生成成功只产生 `review_ready` 草稿，进入 D2 审阅；
- 用户明确点击“启用本周课表”后才变为 `active`；
- 用户离开或拒绝时保留原有效课表；
- 启用后不自动推送手表。

### D2. 本周课表详情

- `review_ready` 状态下明确标识“待启用”，主 CTA 为“启用本周课表”；
- 审阅草稿时展示相对上周和赛季约束的主要变化与生成依据；
- 阶段与周定位；
- 周里程、总时长、力量次数、强度和恢复分布；
- 七天课表，支持同一天多节训练；
- 每节展示类型、距离、时长、主要强度和完成状态；
- 支持进入训练课详情、调整课表、整周推送和周复盘；
- 显示计划与实际匹配结果；
- 推送整周前展示确认清单；
- 推送结果支持全部成功、部分失败、全部失败和逐项重试。

草稿启用、已启用课表查看和历史周查看可以共用页面结构，但操作区必须按状态变化：历史周只读，草稿不可推送。

### D3. 训练课详情

- 名称、训练类型和强度标签；
- 距离、时长、目标配速和心率区间；
- 热身、主训练、放松和执行要点；
- 训前、训中、训后营养建议；
- 可进入 F2 查看当日完整营养建议；
- 力量训练展示具体动作、组 × 次、组间休息和动作说明；
- 支持推送单节、进入训练前页面；
- 休息日或设备不支持的训练明确显示“无需推送”或“暂不可推送”。

### D4. 调整本周课表

- 用户通过教练描述挪课、减量、加课、跳课或生活冲突；
- 顶部固定当前周摘要和被调整对象；
- 教练返回旧 → 新提案；
- 差异解释移动、删除、替换、降量和新增训练；
- 说明对本周总量、关键训练和赛季目标的影响；
- 用户确认后应用；
- 已推送训练变化时，提示手表端残留处理；
- 应用成功返回更新后的 D2。

### D5. 跑者主页（跑者 Tab 根页面）

跑者主页回答三个问题：今天状态如何、今天练什么、最近练得怎样。

通用内容：

- 左上侧边菜单、日期、显式同步入口和最后同步时间；
- 状态摘要，展示恢复、负荷和建议依据；
- 今日活动：只展示上海时区自然日内同步到的活动，不展示昨天或更早记录；
- 今日活动支持 0、1、多条：0 条显示紧凑空状态，1 条直接展示，多条按开始时间排列；
- 每条今日活动展示类型、开始时间、距离、时长、配速或主要指标、训练匹配、AI 点评摘要和反馈状态；
- 今日活动固定显示在“今日训练”上方；先展示今天实际发生的活动，再展示今天计划执行的训练；
- “全部活动”入口；
- 数据过期或同步中状态。

按训练状态变化：

- 无赛季训练计划：展示创建 CTA；
- 有赛季计划但无本周课表：展示赛季摘要和生成本周课表 CTA；
- 有本周课表：只展示“今日训练”模块，直接显示今天的训练、休息或完成状态；
- 首页不展示周完成课次、周里程、进度条、星期选择器、未来关键训练或“查看完整课表”；这些内容统一归属“训练”Tab；
- 同一天多节训练时，在“今日训练”内按时间排列；
- 今日训练可进入 D3；需要查看或管理完整课表时，用户切换到底部“训练”Tab；
- 今日训练已完成：在“今日训练”内显示完成状态，并在“今日活动”中展示匹配活动、AI 点评和填写反馈 CTA；
- 未检测到今日活动时，只显示紧凑纯文本空状态“今天暂无活动”；该区域不提供同步按钮或大面积空状态卡。手表同步统一使用顶部全局同步入口，不能用历史活动填充空白。

有本周课表时，跑者主页的固定内容顺序为：

`状态判断与依据 → 今日活动 → 今日训练`

### D6. 训练前

- 训练名称、强度、距离、时长、配速和心率；
- 热身清单；
- 训前营养和补水提示；
- 手表推送状态；
- 主提示为“请在手表上开始训练”，不伪装成手机开始跑步按钮。

### D7. 训练后反馈

- 触发：新活动同步通知、活动详情、跑者主页今日完成卡；
- RPE 1-10；
- 感受标签，例如腿酸、状态好、天气热、睡眠差、疼痛；
- 可选文字备注；
- 提交失败保留输入并可重试；
- 已提交可再次编辑；
- 疼痛标签触发保守安全提示，不自动修改计划；
- 成功后反馈进入周复盘和后续计划上下文。

### D8. 活动详情

- GPS 轨迹地图；
- 距离、时长、配速、心率、步频、卡路里、爬升和 STRIDE 训练负荷；
- AI 点评、分段、心率、配速等视图；
- 心率和配速时间序列、区间分布和 lap；
- AI 点评标注生成来源和时间，并提供重新生成状态；
- 训练反馈区支持未填写、已填写和编辑；
- 缺少 GPS、时间序列、区间或 lap 时分别显示领域化空状态，不隐藏整个页面。

### D9. 周复盘

- 周完成率、总里程、总时长、状态储备趋势；
- 每节训练的计划、实际、RPE 和 AI 点评；
- 关键洞察和需要关注的问题；
- 是否具备生成下周课表的条件；
- 查看或编辑本周反馈；
- 主 CTA 为“查看下周课表”或“生成下周课表”；
- 从训练中心和周日晚通知均可进入。

### D10. 全部活动

- 从跑者主页进入；
- 按日期倒序分页；
- 支持日期、运动类型和距离范围过滤；
- 每项展示时间、类型、距离、时长、配速、心率和反馈状态；
- 点击进入 D8；
- 支持空状态、过滤无结果、加载更多失败和离线缓存状态。

### D11. 稳定对象标识

- 活动详情和反馈使用不可变 `activity_id` / `label_id`；
- 赛季训练计划使用 `plan_id`，版本使用 `version_id` 或不可变版本号；
- 本周课表使用不可变 `week_id`，展示用日期不是主键；
- 训练课使用不可变 `session_id`，不能只用日期 + 当日序号定位；
- 调课、同日多课或版本变化后，旧 Deep Link 必须解析到原对象或显示“内容已更新”，不能误打开另一训练课。

## 8. 模块 E：数据

### E1. 数据概览（数据 Tab 根页面）

- 静息心率、睡眠 HRV、恢复/疲劳和训练负荷状态；
- 每项显示当前值、相对个体基线、趋势和一句解释；
- 入口：训练负荷、健康趋势、能力雷达、成绩预测、PB；
- 区分手表原始指标与 STRIDE 计算指标；
- 不同手表能力不同，缺失项说明原因。

### E2. 训练负荷

- 短期负荷（acute）、长期负荷（chronic）、状态储备（form）三条曲线和当前数值；
- 状态储备以相对长期负荷的比例分区，不使用固定经典 TSB 阈值；
- 时间范围：30、90、180 天；
- 显示负荷状态解释、连续过载提示和建议依据；
- 数据不足时不使用魔法默认值。

### E3. 健康趋势

- 疲劳、HRV、静息心率、睡眠、训练负荷维度切换；
- 时间范围：7、30、90 天；
- 当前值、7 日均和趋势方向；
- 只展示当前设备真实支持的维度；
- 数据缺口在图表中可见，不用插值伪装完整。

### E4. 能力雷达

- 耐力、速度、阈值、VO2max、经济性、新鲜度等维度；
- 每项显示当前值、历史趋势、定位和提升建议；
- 标注数据更新时间和置信度；
- 缺少校准数据时说明如何通过训练补齐。

### E5. 成绩预测

- 5K、10K、半马、全马预测；
- 主目标距离突出显示预测成绩和配速；
- 当前预测与目标差距；
- VO2max 当前估计与趋势；
- 历史预测变化；
- 标注实时计算或快照来源。

### E6. PB 记录

- 1K、3K、5K、10K、半马、全马；
- 当前 PB、日期、活动入口和历史变化；
- 自动识别来源；
- 新 PB 可在跑者主页展示庆祝状态，但不得阻断主流程。

## 9. 模块 F：营养

### F1. 营养偏好

- 营养建议总开关；
- 饮食类型、过敏和忌口；
- 增肌、减脂、维持或备赛目标；
- 基础代谢和活动消耗建议值，可手动覆盖；

### F2. 每日营养建议

- 热量目标和宏量分配；
- 训练日与休息日差异；

- 标注建议依据和更新时间；
- 未设置偏好时提供清晰设置入口。

### F3. 饮食记录

- 早餐、午餐、晚餐和加餐；

- 日/周摄入与目标差值；
- 与训练负荷并列解释能量平衡；
- 记录失败保留用户输入。

## 10. 模块 G：账号与设置

### G1. 个人中心

- 头像或首字母、昵称、邮箱、累计里程、手表状态；
- 个人信息：复用 B4 并允许编辑；
- 跑步档案：复用 C2；
- 训练目标：查看或发起 C7 调整；
- 营养偏好：进入 F1；
- 通知设置、隐私政策、用户协议；
- 长期偏好与记忆管理；
- 注销账号；
- 退出登录。

### G2. 手表管理

- 显式同步、解绑、重新绑定和未来换表入口；

### G3. 通知

- 通知点击必须进入明确目标页；

### G4. 支持与应用信息

- 意见反馈提供可提交表单，不只显示“即将支持”；
- 常见问题覆盖绑定、同步、课表推送、数据口径和账号；
- 检查更新支持无更新、下载中、失败重试和安装权限状态；
- 关于页展示版本、协议、隐私和联系方式。

### G5. 注销账号

- 解释删除范围、不可恢复内容和保留周期；
- 二次确认并要求输入确认文本；
- 跑团所有者注销时，由服务端检查所有权并提供“转让给指定成员”或“解散跑团”的阻塞处理页；无需进入完整跑团详情；
- 成功后清理本地 token、缓存和推送注册并返回 A1。

## 11. 模块 H：教练

教练是一级 Tab，也是跨模块统一入口。用户无需先理解赛季计划、本周课表或数据页面结构。

### H1. 会话首页

- 最近会话列表；
- 新建会话；
- 每个会话显示标题、更新时间和当前操作对象；
- 首次进入提供状态、训练复盘、调课和计划等建议问题；
- 会话由用户显式新建，不按日期自动创建；
- 每个会话使用稳定 `session_id`，会话详情路由为 `/v2/coach/:sessionId`；
- 跨会话连续性由长期偏好与记忆提供，不把所有主题塞入一条全局线程。

### H2. 对话

- 支持流式文本、Markdown 和结构化卡片；
- 明确显示当前操作对象：某活动、本周课表、赛季训练计划或普通问答；
- 同一会话可跨状态查询、训练复盘和计划调整；
- 支持必要追问，不在信息不足时直接生成修改；
- 错误后可重试，用户原消息不丢失；
- 长对话恢复后保留关键上下文。

### H3. 结构化回答

v1 至少覆盖：

- 指标摘要：当前值、基线、趋势、来源；
- 单次训练复盘：计划、实际、偏差、可能原因、下一步；
- 疲劳分流：证据、短期建议、是否建议调整本周；
- 疼痛分流：风险提示、非医疗诊断声明、停止或就医红旗；
- 本周课表调整提案；
- 赛季训练计划调整提案。

### H4. 提案与应用

- 普通问答升级为计划修改前，先说明将修改哪个对象并征得同意；
- 提案卡展示旧 → 新差异、依据和影响；
- 用户可以继续追问、拒绝或确认；
- 确认后显示应用进度和最终结果；
- 应用失败保留提案并可重试；
- 疼痛和伤病对话默认只读，除非用户明确确认进入计划调整流程。

### H5. 长期偏好与记忆

- 教练可记住用户明确表达的伤病、时间约束、偏好、目标和生活事件；
- 新记忆需要透明回执，用户可纠正；
- 记忆影响后续问答和计划生成；
- 用户可在个人中心查看和删除长期记忆；
- 不把模型推断当成用户确认事实。

### H6. 主动教练提醒

- v1 支持恢复不足、负荷过高、连续漏课和计划已就绪四类主动提醒；
- 提醒必须引用触发依据，并 Deep Link 到数据详情、对应训练课或教练会话；
- 默认只提供建议，不自动修改课表；
- 用户可在通知设置中分别关闭“训练提醒”和“状态预警”；
- 数据不足或设备不支持时不得触发伪精确预警。

## 13. 全局页面状态（HARD）

所有数据页面和异步流程在 Stitch 与 Flutter 中必须显式覆盖适用状态。

### 13.1 基础状态

- 首次加载；
- 已有数据；
- 合法空状态；
- 网络或服务错误；
- 重试中；
- 下拉刷新；
- 离线缓存；
- 数据过期；
- 权限不足或登录过期。

### 13.2 长任务状态

同步和计划生成必须覆盖：

- queued；
- running；
- ready；
- failed；
- interrupted；
- App 重启后恢复；
- 重复请求已存在。

### 13.3 提案状态

- 等待必要信息；
- 生成提案；
- 审阅差异；
- 用户拒绝；
- 用户确认；
- 应用中；
- 应用成功；
- 应用失败可重试；
- 原对象已变化；
- 无有效变化；
- 请求不安全或不合理而被教练拒绝。

### 13.4 手表推送状态

- 可推送；
- 无需推送；
- 当前设备不支持；
- 推送中；
- 全部成功；
- 部分失败；
- 全部失败；
- 单项重试；
- 已推送训练因计划变化可能残留。

### 13.5 可访问性与安全

- 主要触控目标至少 48 logical px；
- 适配系统字体放大和底部安全区；
- 状态不能只依赖颜色；
- 错误信息不得暴露 token、密码、内部路径或原始服务堆栈；
- 疼痛与伤病建议明确不是医疗诊断；
- 用户输入在可恢复失败后不得丢失。

## 17. v1 完成标准

只有同时满足以下条件，手机端才可视为功能完整：

1. 四个底部 Tab 与侧边菜单信息架构一致；
2. A-I 核心页面在 Stitch 有正式设计源；
3. 注册到首次看到活动与状态的流程可完成；
4. 赛季训练计划创建、审阅、启用、调整和历史可完成；
5. 本周课表生成、草稿审阅、启用、推表、训练、反馈、复盘可完成；
6. 跑者主页和训练中心覆盖 §3.2 的关键状态组合；
7. 活动详情可进入反馈，反馈和周复盘有稳定入口；
8. 教练支持会话恢复、结构化回答和确认式写操作；
9. 核心页面覆盖加载、空、错误、离线和过期状态；
10. 通知 Deep Link、手表同步和 Auth 恢复经过真机验证；
11. 用户可管理手表、通知、隐私、退出和注销账号；
12. 所有用户日期按 Asia/Shanghai 展示，数据来源清晰且不伪造。
