---
name: weekly-key-session-skeleton
description: Per-week key-session skeleton rules for the S1 master plan (including the long-vs-short-cycle taper rule).
---
**Weekly key-session skeleton (HARD)**:
- `weeks` must list **every week** between plan.start_date and plan.end_date, **week by week** (week_index incrementing sequentially from 1, week_start written as that week's Monday in ISO date)
- Each entry links to the corresponding phase's `phase_name` and states that week's `target_weekly_km_low` / `target_weekly_km_high` (which should fall within that phase's weekly-volume range; a recovery week takes 70-80% of the phase lower bound)
- `key_sessions[]` lists **only** the key sessions that drive training adaptation or load (long_run / threshold / tempo / interval / vo2max / hill / race_pace / time_trial / tune_up_race / race / strength_key); ordinary easy / aerobic / recovery / commute runs **must not** be listed
- Every non-recovery / non-taper week must have **1-3** key sessions; a race week may list only a single `race` type; a recovery week is allowed 0-1
- Distance-anchored sessions (long_run / race_pace / tune_up_race / race) write `distance_km`; time-anchored sessions (threshold / interval / tempo) write `duration_min`
- Within one week, high-load sessions like threshold / tempo / interval / vo2max / hill / race_pace **must not exceed 2**
- When `profile.weekly_run_days_max <= 3`, key sessions per week **must not exceed 2**; otherwise no more than 3
- **Taper length is set by overall training-cycle length (HARD)**: long cycles (plan total weeks ≥ 12) use a **2-week** taper; short cycles / cup races (plan total weeks < 12) **taper only the race week** (1 week). For those weeks set `is_taper_week=true`, and `target_weekly_km_high` drops ≥ 25% relative to the peak week. (When this conflicts with the per-distance taper length above, this "cycle length" rule takes precedence)
- **Week-over-week volume ramp cap (HARD)**: between adjacent **load weeks** (any week that is NOT a recovery / taper week), `target_weekly_km_high` must not increase by more than **10%** — the ratio `(this load week's km_high) / (the most recent previous non-deload load week's km_high)` **must be ≤ 1.10**. Walk *past* deload weeks when picking the comparison baseline: a `60 → 42 (recovery) → 62` sequence is a normal post-recovery rebound (62 vs the last **load** week 60 = +3%, fine), but `60 → 70` in consecutive load weeks (+17%) is a violation. Build volume gradually (≤ +10%/load-week) and bank the bigger jumps across a recovery week, never week-to-week.
- For a recovery week every 4 weeks set `is_recovery_week=true`, with the corresponding `target_weekly_km_*` taking 70-80% of the phase lower bound
- The longest `long_run.distance_km` in the peak phase must match the target race distance: fm ≥ 28km, hm ≥ 18km, 10k ≥ 10km, 5k ≥ 6km
- **Goal-pace specificity inside the long run (HARD, fm/hm goals)**: an all-easy long run does not build race-specific endurance — embedded goal-pace volume under accumulated fatigue is what converts speed into a finish and pushes the late-race fade point out. So for fm/hm goals the build and peak phases must program **progressive goal-pace volume embedded in the long run**, not merely as separate short race_pace sessions:
  - The goal-pace block run inside / paired with the weekly long run must **escalate week-over-week** through build→peak. By the peak phase target an fm goal-pace (marathon-pace) block of **~22-25km at goal pace inside a 30-34km long run** (hm: ~12-16km at goal pace inside an 18-24km long run).
  - **Smooth, gap-free MP ladder (HARD)**: the embedded MP volume must climb in **steady ~2-4km steps with NO single jump larger than ~5km** and **must not stall then leap** — a ladder that sits in the mid-teens for several weeks and then jumps straight to the peak rehearsal (e.g. `8→10→12→14→14→23`) is forbidden, because cumulative time-at-goal-pace (not one heroic rehearsal) is what makes race pace feel sustainable. Walk every rung: for an fm sub-3 target the MP block should read like **8→12→16→18→20→22(-24)km** across build→peak, hitting the **16km and 18-20km rungs in the build phase** before the maximal ~22-24km peak rehearsal — so the biggest rehearsal is only a 1-2 rung step up, not a 8-9km leap. (Deload weeks interrupt the *calendar* sequence but not the *ladder*: compare each MP long run to the previous MP long run, stepping past recovery/taper weeks, exactly as the volume-ramp rule does.)
  - Represent each such week with BOTH a `long_run` (distance_km = total) and a `race_pace` key session (distance_km = the goal-pace volume) in that week — a peak-phase long run that is pure easy with **no** goal-pace block is not allowed for fm/hm.
  - **Key-session-cap interaction (HARD)**: the `long_run` + `race_pace` pairing is that week's primary quality block and already uses 2 key-session slots, of which `race_pace` is a hard session. To respect the 1-3 key-session cap AND the ≤2 hard-session cap, a week that carries this pairing must **not** also add a separate threshold/tempo/interval if that would push the week past 3 key sessions or 2 hard sessions — the long-run goal-pace block IS the week's hard quality. For athletes with **`weekly_run_days_max ≤ 3`** (key-session cap = 2), do **not** emit a separate `race_pace` session: instead express the goal-pace block **inside the single `long_run`** entry (state the embedded MP volume in its description / the milestone) so the week stays within 2 key sessions. In a `tune_up_race` / `race` week, that race is the week's key session — keep other key sessions minimal.
  - When the fm goal time is **sub-3:00**, the longest peak-phase `long_run.distance_km` must reach **≥ 32km** (this overrides the ≥28km floor above), to rehearse goal pace past the 30km fade point.
- **Peak-block periodization & key-rehearsal placement (HARD, fm/hm goals)**: fitness is not made by the hardest session — it is made by *absorbing* the hardest session. A plan that stacks its biggest rehearsals back-to-back and runs them straight into the taper delivers the athlete to the start line carrying rehearsal fatigue, not race-day freshness. So:
  - **Single peak rehearsal, then absorb (HARD)**: the **single biggest goal-pace rehearsal** (the long run with the largest embedded MP volume — e.g. the 32km/22-24km-MP for fm) must land **3-4 weeks before `race_date`**, and the week **immediately after it must be a recovery / deload week** (`is_recovery_week=true`, 70-80% volume, quality cancelled) so the adaptation is absorbed. **Do not** place the biggest rehearsal in the final load week before the taper (that leaves no absorption — it arrives as fatigue, not fitness).
  - **No consecutive maximal rehearsals (HARD)**: the two largest-MP-volume long runs **must not** fall in **consecutive load weeks**. Separate them by **≥ 1 reduced-load / recovery week**. A monotonic ramp of `…20MP → 22MP → 24MP` in three straight load weeks at peak volume is a violation — escalate, then deload, then peak-rehearse.
  - **Deload before taper (HARD)**: a **recovery / deload week must precede the taper** — never run **4+ consecutive load weeks** (no deload) directly into the taper. Counting the tune-up race week as a load week, the loading region between the last recovery week and the taper must be **≤ 3 weeks**. If the peak phase would otherwise be 3-4 load weeks with no break, insert a deload inside it (peak phases of ≥4 weeks still obey the 3:1 recovery rule).
  - **Peak weekly-volume ceiling (HARD)**: peak `target_weekly_km_high` **must not exceed the athlete's historical `peak_weekly_km_in_window`** (from the history summary) by more than **~5-10%**. A new season is not the time to set an all-time volume record on top of new intensity — cap the peak near the proven ceiling (e.g. history peak 80km → plan peak ≤ ~86km), and buy the goal time with *specificity and freshness*, not unprecedented volume that the athlete's tendons and joints have never absorbed.
- **Strength & durability track (HARD)**: a plan must never be run-only — running volume is bet against a durability ceiling, and for endurance athletes the late-race fade is often a structural (posterior-chain / hip / ankle / tendon) failure, not an aerobic one. So **every plan must program a strength & durability track** alongside the run sessions. Because strength/durability work is **non-running**, it is tracked at the **phase / principles / milestone** level and does **NOT** count toward the weekly 1-3 key-session density cap above:
  - Each phase's `key_session_types` must include a strength/durability entry (e.g. `strength_key`), and `training_principles` must carry one entry describing the durability focus + weekly cadence (typically 2×/week in base/build, reduced in taper).
  - Attach **at least one** strength/durability exit milestone (`type:"strength_test"`).
  - **Injury-aware (HARD)**: when the injected continuity / injury context flags an active injury or a durability weakness, the durability track must name **targeted rehab** for it (e.g. eccentric calf loading for Achilles tendinopathy; posterior-chain / hip / adductor endurance and single-leg stability for late-race fade) — do not emit a generic strength line that ignores a flagged injury.
