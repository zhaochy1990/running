**Per-phase quantifiable milestone (HARD)**:
- Add meaningful `milestones` every ~2-3 weeks, no gap >3 weeks. Each active phase needs one `phase_name`-attached performance checkpoint; numeric targets use `metric` + `target_value` + `comparator`.
- Long-run milestone dates/targets must match the same Monday-Sunday `weeks` entry. Before any `long_run` milestone, copy that week's exact `long_run.distance_km` into `target` and `target_value`. This includes `18km`, `20-26km`, and `28-32km`: a milestone saying those distances must sit in a week whose `key_sessions` has that distance. Do not put big checkpoints on recovery weeks or between skeleton weeks. If the intended long run is on a previous Sunday, set the milestone date to that Sunday; if it belongs to the next Monday-Sunday week, date it inside that week, not the prior Sunday. Example: if `week_start=2026-09-14` has `29km`, milestone date must be 2026-09-14..2026-09-20 (prefer Sunday 2026-09-20), never 2026-09-13.
- Anchor performance targets to the injected **actual personal best (PB)** line, not predictions. Typical metrics: `race_time_s_5k`, `race_time_s_10k`, `race_time_s_hm`, `long_run_distance_km`; embedded MP/HMP distance can stay in `target`. Speed-phase 5K/10K targets stay within a realistic single-cycle ceiling (~5-8%).
- FM/HM build/peak milestones are goal-pace endurance checks: `long_run` total + embedded MP/HMP volume. Tune-up races/time trials map to `test_run`; HM/10K gates must validate the race goal and state PB-ratio/Riegel basis. Aggressive A gates are strict: for FM A=2:50, only HM `<=1:24:30` or 10K `<=37:45` can contribute to A, with 31-32km/22-24kmMP (30km only under risk/history/ramp cap) + VO2/HR/RPE + Achilles normal. HM `<=1:25:30`, bare `<=1:25`, or 10K `>=38:00` is observation/B only; never label `A通道` or emit `<=1:27:30`.
- Aggressive HM near 9% faster than PB (HM 1:27:42 -> sub-1:20): `10K<=39:00` is only a B/observation gate. If a 10K tune-up opens A, require `10K<=37:00`; treat `10K<=37:45`/`<=38:00` as observation/B+ only. Also require a 20-22km long run with 12-16km at HMP + normal HR/RPE, and put A/B/C plus gates in the race milestone, not just `A<1:20`.
- If recent race data is stale (>12 months) or current pace is unknown, add a mid-base/build `test_run` milestone as 10K or HM tune-up, not only 5K; target says it recalibrates later training paces.
- Sparse watch/app migration + credible advanced self-report is not stale fitness: no `test_run`/`time_trial` to re-prove PRs; use long-run/MP gates.
- Borderline-aggressive FM just above realism threshold (PB 3:10 -> goal 2:45, ~13%): keep user goal as gated A, train paces by safer B, and require HM <=1:18:00 or 10K <=36:00 plus largest legal rehearsal (30-32km MP when legal; otherwise 28-29km at <= min(historical_peak * 1.10 + 2km, historical_peak + 7km)) with normal HR/RPE. Add a mid-cycle `test_run` milestone carrying this strict A gate; if a tune-up is easier, label it observation/B only. Do not loosen the A gate to HM 1:18:30 or slower. In A/B/C wording, set C=PB/no-regression finish (for PB 3:10, C=PB/finish around 3:10), not sub-3.
- If user gives A/B/C goals, race milestone `target` must state all thresholds and how A opens. Compact example: `A=2:50仅HM<=1:24:30或10K<=37:45 + 31-32km/22-24kmMP + 跟腱/VO2/HR/RPE全过时开放；B=2:52-2:53；C<3h`. Never leave bare `A/B/C` times.
- If user names decision signals (VO2max plateau, high RHR, altitude adaptation, Achilles pain), include them in A-gate wording or nearby principle/monitoring trigger. For zhaochaoyi S1, the first A-gate principle/race milestone must spell out: HM is only an observation gate, A opens only with largest legal MP rehearsal (prefer 31km at 89-91km high; 32km only at >=92km; 30km only with explicit risk cap) + VO2max + HR/RPE + Achilles response. Never summarize as just `HM+31km过关` or HM alone.
- Body composition is secondary: use `body_composition` / `weight_kg` / `body_fat_pct` only with a baseline or explicit start->target path, at most once per plan; never replace performance checkpoints or emit “maintain current value”.
- Add at least one `strength_test` durability milestone. If continuity/injury names weakness, target it specifically (Achilles eccentric/isometric calf capacity, quad eccentric/step-down durability, hip/glute/ITB control), not generic strength. If linked risks include Achilles + ITB/hip-glute + quad durability, milestone or adjacent principle must name all three compactly: `跟腱等长/离心 + 髋臀稳定 + 股四头离心/下台阶`; do not collapse to calf raises or generic strength.
